JPQL에서 매핑되지 않은 관계의 LEFT JOIN 한계

들어가며

JPA를 쓰다 보면 이런 상황을 만납니다.

▎ "Call 테이블과 CallEmotionResult 테이블을 LEFT JOIN해서 감정 분석이 없는
▎ 통화도 포함시키고 싶다."

SQL로는 간단합니다.

SELECT c._, cer._
FROM calls c
LEFT JOIN call_emotion_results cer ON c.id = cer.call_id AND cer.user_id = ?
WHERE c.user1_id = ? OR c.user2_id = ?

그런데 JPQL로 이걸 그대로 옮기려 하면 예상치 못한 벽을 만납니다.

---

JPQL LEFT JOIN의 제약

JPQL에서 JOIN은 엔티티 간에 선언된 연관관계(@OneToMany, @ManyToOne 등)를
따라가는 것입니다.

// ✅ JPQL JOIN이 가능한 경우 — 연관관계가 선언되어 있음
@Entity
public class Call {
@OneToMany(mappedBy = "call")
private List<CallEmotionResult> emotionResults;
}

이 경우엔 아래처럼 JPQL LEFT JOIN을 쓸 수 있습니다.

@Query("""
SELECT c FROM Call c
LEFT JOIN c.emotionResults cer
WHERE (c.user1.id = :userId OR c.user2.id = :userId)
""")
List<Call> findWithEmotionResults(@Param("userId") Long userId);

하지만 연관관계가 없다면 어떻게 될까요?

---

연관관계를 선언하지 않은 이유

Call 엔티티에 @OneToMany List<CallEmotionResult>를 붙이지 않은 데는 이유가
있습니다.

1. 불필요한 결합도

Call은 통화 진행 자체를 담당하는 엔티티이고, CallEmotionResult는 분석 결과를
저장하는 별도 도메인입니다. 통화 엔티티가 분석 결과 컬렉션을 직접 들고 있으면,
통화 로직을 다룰 때마다 분석 결과가 함께 로드될 수 있습니다.

2. N+1 및 성능 우려

@OneToMany를 달면 기본이 LAZY이지만, JPQL fetch join 없이 접근하면 N+1 쿼리가
발생합니다. 실수할 여지가 늘어납니다.

3. 방향성

Call이 CallEmotionResult를 알아야 하는 이유가 없습니다. 분석 결과 조회는 분석
도메인의 책임입니다.

이처럼 의도적으로 JPA 연관관계를 끊어두는 선택은 실무에서 흔합니다. 그런데 이
선택의 대가로 JPQL LEFT JOIN을 쓸 수 없게 됩니다.

---

JPQL에서 임의 조건 JOIN을 시도하면

연관관계 없이 JPQL에서 LEFT JOIN을 시도하면 어떻게 될까요?

// ❌ 컴파일은 되지만 실행 시 오류
@Query("""
SELECT c FROM Call c
LEFT JOIN CallEmotionResult cer ON c.id = cer.callId AND cer.userId =
:userId
WHERE c.user1.id = :userId OR c.user2.id = :userId
""")

Hibernate 5 이하에서는 위 문법 자체가 지원되지 않아 파싱 오류가 납니다.
Hibernate 6+에서는 임의 조건 JOIN(ad hoc join) 문법이 도입되었지만, Spring Boot

- Hibernate 버전 조합에 따라 동작이 다르고, 여전히 엔티티 연관관계 없이는
  제약이 있습니다.

근본적으로 JPQL은 객체 그래프를 탐색하는 언어이기 때문에, SQL처럼 임의의 컬럼을
기준으로 자유롭게 JOIN하는 것을 목적으로 설계되지 않았습니다.

---

해결 방법 1: 네이티브 쿼리

가장 직접적인 해결책은 @Query(nativeQuery = true)로 SQL을 그대로 쓰는 것입니다.

@Query(value = """
SELECT c.id, c.start_at, c.end_at, c.duration_seconds,
cer.all_scores, cer.dominant_emotion
FROM calls c
LEFT JOIN call_emotion_results cer
ON c.id = cer.call_id AND cer.user_id = :userId
WHERE (c.user1_id = :userId OR c.user2_id = :userId)
AND c.call_status = 'COMPLETED'
""", nativeQuery = true)
List<Object[]> findMonthlyRaw(@Param("userId") Long userId);

단점:

- 반환 타입이 Object[] 또는 Projection 인터페이스로 제한됨
- DB 방언에 의존 → DB 교체 시 쿼리 수정 필요
- 컴파일 타임에 오류를 잡을 수 없음
- JSON 컬럼(all_scores)을 별도로 역직렬화해야 함

---

해결 방법 2: 두 쿼리 분리 후 Java에서 병합

이 프로젝트에서 선택한 방식입니다. 각 레포지토리에서 별도로 조회하고, 서비스
레이어에서 합칩니다.

Step 1 — Call 조회 (감정 분석 여부 무관)

// CallRepository
@Query("""
SELECT c FROM Call c
WHERE (c.user1.id = :userId OR c.user2.id = :userId)
AND c.callStatus = 'COMPLETED'
AND c.endAt >= :startDate AND c.endAt < :endDate
ORDER BY c.endAt, c.id
""")
List<Call> findCompletedCallsByUserInDateRange(
@Param("userId") Long userId,
@Param("startDate") LocalDateTime startDate,
@Param("endDate") LocalDateTime endDate
);

Step 2 — 감정 분석 결과 조회

// CallEmotionResultRepository
List<CallEmotionResult> findByCallIdInAndUserId(List<Long> callIds, Long
userId);

Step 3 — 서비스에서 병합

public List<DailyEmotionSummaryResponse> getMonthlyEmotionResults(
Long userId, int year, int month) {

      LocalDateTime startDate = LocalDate.of(year, month, 1).atStartOfDay();
      LocalDateTime endDate = startDate.plusMonths(1);

      // 쿼리 1: 완료된 통화 전체
      List<Call> calls = callRepository.findCompletedCallsByUserInDateRange(
              userId, startDate, endDate);

      List<Long> callIds = calls.stream().map(Call::getId).toList();

      // 쿼리 2: 감정 분석 결과
      Map<Long, CallEmotionResult> resultByCallId =
          callEmotionResultRepository.findByCallIdInAndUserId(callIds, userId)
              .stream()
              .collect(Collectors.toMap(r -> r.getCallId(), r -> r));

      // Java에서 LEFT JOIN과 동일한 효과: 분석 없는 통화는 null
      return buildDailySummaries(calls, resultByCallId);

}

private List<DailyEmotionSummaryResponse> buildDailySummaries(
List<Call> calls,
Map<Long, CallEmotionResult> resultByCallId) {

      return calls.stream()
          .collect(Collectors.groupingBy(
              c -> c.getEndAt().toLocalDate(),
              LinkedHashMap::new,
              Collectors.toList()
          ))
          .entrySet().stream()
          .map(entry -> buildDailySummary(entry.getKey(), entry.getValue(),

resultByCallId))
.toList();
}

감정 분석이 없는 통화는 resultByCallId.get(callId)가 null을 반환하므로,
자연스럽게 LEFT JOIN과 동일한 결과를 얻습니다.

두 접근 방식 비교

┌───────────────────┬─────────────────────────┬─────────────────────────┐
│ 항목 │ DB JOIN (네이티브 쿼리) │ 두 쿼리 + Java 병합 │
├───────────────────┼─────────────────────────┼─────────────────────────┤
│ 쿼리 횟수 │ 1회 │ 2회 │
├───────────────────┼─────────────────────────┼─────────────────────────┤
│ 반환 타입 │ Object[], Projection │ 엔티티 (타입 안전) │
├───────────────────┼─────────────────────────┼─────────────────────────┤
│ DB 방언 의존 │ 있음 │ 없음 │
├───────────────────┼─────────────────────────┼─────────────────────────┤
│ 컬럼 추가 시 영향 │ 쿼리 수정 필요 │ 엔티티 필드 추가만 │
├───────────────────┼─────────────────────────┼─────────────────────────┤
│ N 증가 시 │ DB가 최적화 │ 메모리 사용 증가 │
├───────────────────┼─────────────────────────┼─────────────────────────┤
│ 테스트 용이성 │ 낮음 (통합 테스트 필요) │ 높음 (단위 테스트 가능) │
└───────────────────┴─────────────────────────┴─────────────────────────┘

---

어떤 경우에 어떤 방식을 선택할까?

두 쿼리 분리 + Java 병합이 유리한 경우

- 조회 데이터 양이 수백~수천 건 수준 (한 유저의 월별 통화 이력 등)
- 각 쿼리가 인덱스를 잘 탄다
- 이미 분리된 도메인을 유지하고 싶다
- 반환 타입이 엔티티여야 JPA 매핑이 자동으로 동작해야 한다

네이티브 쿼리 혹은 엔티티 연관관계 추가가 유리한 경우

- 조인 대상이 수십만 건 이상이고 DB 수준의 최적화가 필요하다
- 집계(GROUP BY, COUNT 등)가 복잡해서 애플리케이션에서 처리하기 어렵다
- 두 테이블이 항상 함께 쓰이고 분리 조회의 이점이 없다

실무에서 마주치는 유사한 패턴
이 패턴은 "느슨하게 결합된 두 도메인을 조회할 때" 자주 등장합니다.

- Order ↔ Review: 주문에 리뷰가 없을 수도 있고, 리뷰 도메인이 주문 도메인과
  JPA로 묶이지 않을 때
- User ↔ PushNotificationToken: 유저가 토큰을 가질 수도, 없을 수도 있을 때 - Post ↔ ReadHistory: 읽음 여부를 별도 테이블에서 관리할 때

                    공통점은 "한쪽이 없어도 다른 쪽은 조회되어야 한다(LEFT JOIN 의미)"는 요구와 "두   도메인을 JPA로 결합하기 싫다"는 설계 원칙이 충돌하는 상황입니다.

이때 두 쿼리 분리 + Map으로 매칭하는 패턴은 간단하면서도 타입 안전성을 유지할 수 있는 실용적인 선택입니다.

---

마무리
JPQL LEFT JOIN은 편리하지만 엔티티 연관관계가 선언되어 있어야만 사용할 수
있습니다. 도메인 설계 원칙에 따라 연관관계를 의도적으로 제거했다면, 두 쿼리를 분리하고 서비스 레이어에서 병합하는 방식으로 동일한 효과를 낼 수 있습니다.

이 트레이드오프를 이해하고 있으면, JPA를 사용하면서 "왜 SQL로는 되는데 JPQL로는
안 되지?"라는 상황에서 빠르게 원인을 파악하고 설계 결정을 내릴 수 있습니다.
